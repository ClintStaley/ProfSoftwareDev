1. Explore the existing PhysicsCompetition repo, and answer the following. (If you do this on a Unix or Gitbash commandline, you may find the wc command useful.):


  • How many commits have been made to PhysicsCompetition? (Note you can get a listing of all commits, one per line.)

  • When was the first?

  • Could you determine this even if your machine were not on the internet? Why?

  • How much space is used by each commit, on average? (The size of the .git directory, divided by number of commits, would be a good way to estimate this.) Does Git really store each commit in its entirety?


2. Git often uses 7-letter hashcode prefixes to identify a commit. Is this potentially ambiguous? What are the odds that two commits might have the same 7-letter prefix?


3. What does git do when competing edits apply to a binary file, e.g. a word doc?


4. Does git force a hand-merge when two different commits edit the same file, or can the merge be automatic if the edits are not overlapping? Test to find out how close two edits can be and still result in automatic merge.


5. A commit "ome fix comments" was made early in the history of the project. What files/lines did it change? Why?


6. How many commits were made on Aug 25, 2017?


5. Do the following Git exercise, logging the commands you ran as a document to submit:


  1. Create a repo GitExercise in your github account.

  2. Clone GitExercise onto your local machine, but do not incorporate your password into the remote URL. Instead, set up a credential cache system using git config --global to set a credential.helper property. (See Credential Storage in the Git text.) Set it for 1 hour timeout, and confirm that you only need to enter your password once per hour.

  3. Copy the contents of the Maven project you did in an earlier assignment into GitExercise. We'll use it as a small app to manage via Git.

  4. Set up git so it ignores the entire target directory, plus any directory named "Archive" anywhere in the tree. Create an Archive directory under src and put a small file in it.

  5. With a single command, add all files (pom.xml, src/**) to the repository.

  6. Confirm that all files are now staged for commit, except those you told git to ignore.

  7. Commit with the message "Initial commit" and push to the remote repo.

  8. Create, stage and commit a new file Sample under src, with 10 distinct lines of text.

  9. Use git branch to create a new branch SampleMod. Check out the branch

  10. In SampleMod, change line 10 of Sample and commit the change.

  11. In master, change line 1 of Sample and commit the change

  12. Switch between SampleMod and master, and use git log to see the different commit history.

  13. In SampleMod, use git diff to find the difference between your most recent commit and that before it. Find a couple of different ways to describe those two commits in your diff command.

  14. Use git diff to show the differences between the head of SampleMod and the head of master

  15. Merge SampleMod into master

  16. Make another modification to Sample in the SampleMod branch

  17. Delete the SampleMod branch and get rejected due to potential data loss

  18. Revoke the most recent commit to SampleMod.

  19. Successfully delete SampleMod now.

  20. In master branch "accidentally" delete file Sample

  21. Restore file Sample from the repository.

  22. Modify Sample by adding one line, and add it to the staging area. Confirm you did so via git status

  23. Remove the modified Sample from the staging area, retaining the modification in your working tree only. (What level of "reset" is this?)

  24. Now add and commit the modified Sample. Confirm this via git status and git log

  25. Now, get rid of that last commit entirely, returning exactly to the pre-commit state.

  26. Now, *using only git reset* restore that commit.

  27. Now, undo that commit, including reverting your working tree to the state before step 22. Check that this has happened by examining Sample and git log

  28. And, finally, bring back the reverted commit, including your working tree, using git reset only.

  29. Push all this work to github.
